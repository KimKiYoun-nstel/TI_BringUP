# Docs Guide

`docs/`는 TI 보드 브링업 과정에서 나온 지식, 로그, 결정, 작업 항목을 재사용 가능하게 축적하는 공간입니다.

이 가이드는 새 문서를 추가할 때 "이 문서를 어디에 둬야 하는가?"를 빠르게 판단하기 위한 기준입니다.

## 분류 원칙

- 문서는 "어떻게 만들어졌는가"보다 "나중에 무엇을 찾기 위해 다시 볼 것인가"를 기준으로 분류합니다.
- 하나의 문서에는 하나의 주 분류만 둡니다. 성격이 섞이면 주 목적지에 두고 다른 문서에서 링크로 연결합니다.
- 원본 증적과 정리 문서는 분리합니다.
  - 원본 로그, 명령 출력, 실패 재현 기록은 `bringup-logs/`
  - 정리된 해설, 절차, 요약은 `boards/`, `common/`, `setup/`, `outputs/`
- 보드별로 반복 참조할 **해결된 이슈 히스토리**는 `boards/<board-name>/issues/` 처럼 보드 폴더 아래에 둡니다.
- 보드에 강하게 종속되면 `boards/<board-name>/`, 보드와 무관하게 재사용되면 `common/` 또는 `setup/`에 둡니다.
- 확정된 결론은 `decisions/`, 아직 탐색 중인 내용은 `research/`에 둡니다.
- 진행 관리 문서는 `tasks/`, 재사용 서식은 `templates/`에 둡니다.
- 새로 작성하거나 수정하는 Markdown 문서는 기본적으로 한글로 작성합니다.
- 코드, 경로, 명령어, 로그 원문은 필요 시 원문 그대로 유지하되 설명과 판단은 한글로 정리합니다.

## 빠른 판단 순서

1. 세션 로그, 부팅 로그, 명령 결과, 실패 재현 기록인가?
   - `bringup-logs/`
2. 특정 보드 하나의 특성, 설정, 이슈, 검증 메모인가?
   - `boards/<board-name>/`
   - 해결된 이슈의 짧은 히스토리는 `boards/<board-name>/issues/`
3. SDK, 툴체인, 호스트 환경, 빌드 환경 준비 절차인가?
   - `setup/`
4. 여러 보드에서 공통으로 쓰는 개념, 흐름, 용어, 범용 팁인가?
   - `common/`
5. 아직 결론이 나지 않은 조사/비교/가설 정리인가?
   - `research/`
6. 앞으로 기준으로 삼을 결정과 그 근거인가?
   - `decisions/`
7. 해야 할 일, 일정, 추적 보드, 다음 액션인가?
   - `tasks/`
8. 반복 작성용 문서 뼈대인가?
   - `templates/`
9. 다른 문서들을 바탕으로 정리한 체크리스트, 비교표, 요약 산출물인가?
   - `outputs/`
10. 두 개 이상 보드를 묶는 실험 단위인가?
   - 루트 `projects/<project>/`

## 폴더별 목적

### `common/`

보드 종류와 무관하게 반복 참조하는 배경지식과 범용 레퍼런스를 둡니다.

예:

- 부트 플로우 설명
- BSP 용어 정리
- 공통 이미지 작성 절차
- 공통 issue resolution workflow

### `boards/`

보드별로 오래 유지할 메모를 둡니다. 폴더명은 보드 식별자 기준으로 유지합니다.

예:

- 보드 개요
- 보드별 UART/부트 모드/전원 메모
- 특정 보드에서만 유효한 검증 결과
- 특정 보드용 사용 가이드
- 보드별 해결 이슈 히스토리

권장 구조:

```text
boards/
  SK-AM64B/
    README.md
    issues/
      README.md
      YYYY-MM-DD_<topic>.md
    power-notes.md
    uart-console-windows.md
  TMDS64EVM/
    README.md
```

### `bringup-logs/`

날짜 기준으로 남기는 실행 기록 저장소입니다. 원본성을 우선합니다.

예:

- 첫 SD 부팅 로그
- U-Boot 빌드 후 부팅 검증 로그
- 실패 증상과 재현 절차
- 첨부 로그 파일(`.rtf`, `.txt`)

권장 파일명:

```text
YYYY-MM-DD_<board>_<topic>.md
```

### `setup/`

개발 PC, SDK, 빌드 도구, 플래싱/이미지 생성 환경처럼 "준비 절차"가 중심인 문서를 둡니다.

예:

- Processor SDK 초기 설치
- U-Boot standalone build 환경 구성
- 호스트 툴 설치 및 검증

### `research/`

탐색 중인 메모를 둡니다. 아직 결론이 아니며, 비교와 가설이 살아 있는 문서가 여기에 해당합니다.

예:

- TI 문서 비교 메모
- 부팅 경로 후보 분석
- 원인 가설과 확인 필요 항목

### `decisions/`

팀 또는 저장소 기준으로 채택한 판단을 기록합니다. 나중에 "왜 이렇게 했는가?"를 설명하는 기준점입니다.

예:

- 기본 부팅 매체 선정 이유
- SDK 버전 선택 이유
- 로그 보관 정책

### `tasks/`

현재 진행 중인 액션 아이템과 추적 문서를 둡니다. 완료 후 장기 가치가 생기면 다른 폴더의 정리 문서로 승격할 수 있습니다.

예:

- 다음 작업 목록
- 작업 보드
- 주간 우선순위

### `templates/`

반복 작성 시 복사해서 쓰는 템플릿을 둡니다.

예:

- bring-up log 템플릿
- decision 템플릿
- research note 템플릿
- issue history 템플릿

### `outputs/`

다른 문서에서 끌어온 내용을 재구성한 "공유용 산출물"을 둡니다. 원본 로그나 러프 노트보다 한 단계 정제된 문서입니다.

예:

- 보드 비교표
- 커스텀 보드 포팅 체크리스트
- SDK 셋업 가이드 요약본

## 자주 헷갈리는 경우

- 보드별 사용법인데 명령 로그도 많다
  - 원본 로그는 `bringup-logs/`, 최종 정리본은 `boards/<board-name>/`
- 공통 개념 설명인데 이번 작업에서 처음 정리했다
  - 특정 세션에서 시작했더라도 재사용 범위가 넓으면 `common/`
- 설치 절차 안에 특정 보드 예시가 들어간다
  - 문서의 중심이 환경 준비면 `setup/`, 보드 차이는 예시로만 남깁니다
- 아직 조사 중이지만 사실상 방향을 정했다
  - 기준으로 삼을 결정이면 `decisions/`, 검토 흔적은 `research/`에 남깁니다
- 보드 두 개 이상을 함께 다루는 실험인데 보드 메모와 절차가 섞인다
  - 고정 보드 사실은 `board/`, 장기 해설은 `docs/boards/`, 실험 절차와 결과는 `projects/<project>/`로 분리합니다

## 새 카테고리 추가 기준

기존 폴더 안에서 문서 수가 반복적으로 늘어나고, 문서의 목적이 명확히 구분되며, 앞으로도 계속 누적될 성격이라면 새 하위 폴더를 추가할 수 있습니다.

새 카테고리를 만들 때는 다음 둘을 함께 갱신합니다.

- 이 문서(`docs/README.md`)
- 루트 `README.md`의 `docs` 구조 설명

## 커밋 규칙

- 문서 커밋 메시지는 한글로 작성합니다.
- 가능하면 `docs:` 접두어 뒤에 변경 목적을 짧게 씁니다.

예:

```text
docs: 문서 분류 지침 추가
docs: SK-AM64B OSPI 부팅 검증 기록
docs: SDK12 U-Boot standalone 빌드 절차 정리
```
